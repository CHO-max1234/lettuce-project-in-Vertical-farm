260422 오늘 할 일 - 집 CODEX 인계
================================

작성 목적
---------

사무실에서 오늘 작업을 못 했으므로, 집 CODEX가 바로 이어서 작업할 수 있도록 현재 판단과 오늘 해야 할 일을 정리한다.

중요한 태도:
- 이전 GPT 통찰을 전부 그대로 흡수하지 말 것.
- 받아들일 것, 의문 가질 것, 바로 검증할 것을 구분할 것.
- 지금부터는 OCR 엔진 실험이 아니라 real_bed_id 복구 규칙 설계로 봐야 한다.


현재까지 파악한 핵심
-------------------

1. Tesseract는 더 볼 필요가 거의 없다.

사용자 판단:
- "260421 OCR 맛보기"에서 Tesseract는 처참하게 실패했다.
- 따라서 Tesseract를 다시 살리는 데 시간을 쓰지 말 것.
- EasyOCR 딥러닝 함수 하나를 파이프라인에 추가하는 방향으로 간다.

CODEX 판단:
- 이 판단을 받아들인다.
- 현재 50장 테스트에서 EasyOCR은 human 기준 48/50 성공, 즉 96% 수준이다.
- 병목은 OCR 엔진이 아니라 crop 품질, 결과 저장 스키마, OCR 결과를 offset/sequence 복구 규칙으로 넘기는 후처리다.


2. 50장 CSV에서 가장 강한 통찰은 +64 mod 95 offset이다.

입력 CSV:
G:\다른 컴퓨터\내 컴퓨터\새 폴더\양상추사진\양상추_테라웨이브\양상추 날짜\RGB_정면\4. 결과 출력 시각화\2작기 error\260421 piece crop_ex\VER2_Gemini\260422_bed_ocr_results_crop.csv

주의:
- 이 CSV에는 piece_crop_test_montage.jpg 행이 들어 있다.
- 실제 이미지 샘플 50장 분석에서는 montage 행을 빼야 한다.

현재 50장 기준:
- human 기준 정답: 48/50
- 실패 2건:
  - bed01_20260328_185534_cam1_piece.jpg : 실제 71, OCR -1
  - bed01_20260328_191502_cam1_piece.jpg : 실제 81, OCR 8, confidence 1.0

offset 계산:
- cyclic_offset_95 = (real_or_human_bed_num - saved_num) mod 95
- 1~95 순환 기준으로 계산한다.

bed01 제외 결과:
- offset 64: 26건, 20260308~20260329, bed30/bed50/bed70/bed90
- offset 0 : 14건, 20260402~20260411, bed30/bed50/bed70/bed90

즉 샘플 안에서는 아래가 깔끔하게 맞는다.
- saved_bed 30 -> real 94
- saved_bed 50 -> real 19
- saved_bed 70 -> real 39
- saved_bed 90 -> real 59

해석:
- 3월 구간의 일부 saved_bed는 랜덤 오염이 아니라 전역 cyclic offset을 가진다.
- 4월 초, 대략 20260402~20260403 근처부터 saved_bed = real_bed로 정상화된 듯하다.


3. bed01은 일반 규칙에 넣으면 안 된다.

bed01의 human 값:
- 3, 18, 93, 63, 11, 73, 71, 81, 32

판단:
- bed01은 단순 offset 규칙이 아니다.
- fallback bucket, contaminated bucket, 저장명 오류가 몰린 통일 가능성이 높다.
- 다른 sampled bed들보다 bed01 cam1 이미지 수가 압도적으로 많았던 점도 수상하다.
- bed01은 별도 QC 그룹으로 빼고, 일반 offset 추정에서 제외하는 것이 안전하다.


받아들일 것
-----------

1. EasyOCR을 주력으로 쓴다.
- Tesseract 재검토는 하지 않는다.
- B/C/D/E는 Tesseract 살리기용 성격이 강했다.
- EasyOCR 입력은 A crop 또는 B crop 중 실제 성능과 안정성을 보고 결정한다.

2. +64 mod 95 offset은 강한 후보 규칙이다.
- 단, 현재는 bed30/50/70/90 샘플에서 강한 것.
- 전체 95개 bed에 일반화하기 전에는 전체 7,342장 OCR 결과로 재검증해야 한다.

3. confidence 단독 기준은 금지한다.
- 81 -> 8인데 confidence 1.0인 케이스가 있다.
- confidence가 높아도 한 자리 결과면 suspicious로 봐야 한다.

4. bed01은 별도 처리한다.
- 일반 offset 규칙의 anchor로 쓰지 않는다.
- contaminated/fallback bucket 후보로 따로 분석한다.

5. 오늘은 50장에 더 머무르지 말고 전체 적용 준비로 넘어간다.
- 단, 전체 적용은 "최종 확정"이 아니라 "전수 OCR + 규칙 탐색용 wide table 생성"으로 봐야 한다.


의문 가질 것
-------------

1. "3월 전체가 +64 offset"인지는 아직 확정 금지.
- 현재는 샘플 bed30/50/70/90에서만 강하다.
- 전체 bed에서 같은 offset인지, 일부 bed만 그런지 확인해야 한다.

2. 전환 시점은 아직 정확하지 않다.
- 샘플상 20260402~20260403 근처.
- 20260330, 20260331, 20260401 자료가 있어야 경계를 정확히 잡을 수 있다.

3. EasyOCR 결과를 바로 real_bed_id로 확정하면 안 된다.
- 순서:
  1) 전체 OCR 실행
  2) offset 규칙 추정
  3) QC 라벨링
  4) inferred_real_bed_id 부여
  5) 필요 시 manual review

4. OCR 실패 원인은 엔진보다 crop/후처리일 수 있다.
- 실패 2건은 B crop으로 보면 번호가 선명하다.
- A crop이 너무 넓어서 배경/기계 구조물이 OCR에 영향을 줬을 가능성이 있다.
- 81 -> 8은 top-1 저장 방식이나 후보 선택 로직 문제일 수도 있다.


오늘 실제 작업 순서
-------------------

1. 현재 50장 CSV를 정리한다.

해야 할 일:
- montage 행 제거
- image_name에서 saved_bed, saved_num, date, time 파싱
- Human 값을 manual_real_bed_id 또는 human_real_bed_id로 명확히 이름 변경
- 기존 "bools to name", "bools to human"은 컬럼명 정리

기본 컬럼에 추가 
- date
- time
- ocr_all candidates
- ocr_preprocess version
- ocr_text length
- ocr_is_single_digit
- anchor candidate 여부
- plate_qc
- final_anchor_flag

qc_status 초안:
- anchor_good: 2자리 이상, 1~95 범위, 구간 offset과 일치, bed01 아님
- needs_review: confidence 낮음, 한 자리, bed01, offset 불일치
- ocr_fail: detected_num 없음 또는 -1


2. 50장 기준 offset 표를 만든다.

반드시 뽑을 표:

표 A. 전체 offset 분포
- offset
- count
- saved_beds
- date_min
- date_max

표 B. bed01 제외 offset 분포
- offset 64와 offset 0이 얼마나 깔끔한지 확인

표 C. 날짜별 offset 분포
- date
- offset
- count
- examples, 예: 30->94

표 D. saved_bed별 real/human 변화
- saved_bed
- date_min
- date_max
- unique_human_values
- first_real
- last_real
- switch_candidate_date


3. 전체 7,342장 complete triplet에 EasyOCR 적용 준비를 한다.

기존 manifest 관련 파일:
G:\내 드라이브\양상추 분류모델\practice code\양상추 정면 2작기(260306-260430)\OCR\00_build_triplet_manifest.py

handoff summary에 따르면 전체 manifest 결과:
- parsed_image_files = 23,348
- triplet_rows_all = 8,367
- triplet_rows_complete = 7,342
- complete_only 안에서 cam1_path 누락 = 0
- saved_bed 종류 = 96개

전체 OCR 입력은 complete triplet의 cam1_path를 사용한다.
cam0/cam2는 같은 triplet의 cam1에서 복구한 real_bed_id를 상속받는다.


4. EasyOCR 전체 실행 결과는 "얇은 CSV"로 만들지 말 것.

기존 CSV처럼 detected_num/confidence만 남기면 나중에 원인 분석이 불가능하다.

전체 OCR 결과 추천 컬럼:
- triplet_key
- saved_bed
- saved_num
- date
- time
- cam1_path
- piece_crop_path
- plate_crop_path
- preprocess_version
- ocr_raw_text
- ocr_all_candidates
- ocr_best_text
- ocr_best_digits
- ocr_best_conf
- ocr_text_len
- is_single_digit
- valid_range_1_95
- same_as_saved
- cyclic_offset_95
- rule_applied
- anchor_flag
- qc_status
- inferred_real_bed_id
- fill_method
- manual_real_bed_id
- note

중요:
- EasyOCR 후보 전체를 남길 것.
- 최소한 JSON 문자열로라도 all_candidates를 저장할 것.
- top-1 detected_num만 저장하지 말 것.


5. 전체 결과에서 offset 규칙을 자동 탐지한다.

기본 아이디어:
- 날짜 또는 날짜 구간별로 묶는다.
- saved_bed별 detected_num을 모은다.
- offset = (detected_num - saved_num) mod 95 계산.
- 동일 offset이 여러 saved_bed에서 반복되면 그 날짜/구간의 offset 후보로 표시한다.

규칙 후보 예:
- pre_20260402 또는 pre_20260403: offset 64
- post_20260403: offset 0
- bed01: contaminated bucket, rule 제외

단, 전체 결과가 나오기 전까지 경계 날짜는 확정하지 말 것.


6. inferred_real_bed_id 부여는 QC와 분리한다.

좋은 구조:
- detected_num: OCR이 읽은 숫자
- inferred_real_bed_id: 규칙을 적용해 추론한 최종 후보
- rule_applied: 어떤 규칙을 썼는지
- anchor_flag: 이 행을 offset 추정 anchor로 써도 되는지
- qc_status: 신뢰 상태
- manual_real_bed_id: 사람이 최종 수정할 공간

절대 하지 말 것:
- OCR 결과를 바로 최종 real_bed_id로 확정
- confidence만 보고 확정
- bed01을 일반 offset anchor로 사용


실패 2건 재확인 메모
-------------------

실패 2건:
- bed01_20260328_185534_cam1_piece.jpg: 71인데 -1
- bed01_20260328_191502_cam1_piece.jpg: 81인데 8

B crop 이미지를 보면 두 번호 모두 눈으로 선명하다.
따라서 원본 자체 문제보다는 A crop이 넓거나, EasyOCR 입력/후보 선택 방식 문제일 가능성이 크다.

집 CODEX가 가능하면:
- A crop EasyOCR 결과와 B crop EasyOCR 결과를 비교해 볼 것.
- 특히 71/81 두 장이 B crop에서 살아나는지 확인할 것.
- 살아난다면 전체 OCR 입력을 A crop이 아니라 B plate crop으로 바꾸는 방안을 우선 고려.


최종 오늘 목표
--------------

오늘 목표는 모델 성능 개선이 아니다.

목표:
1. 50장 결과를 정리해서 +64 offset 후보와 bed01 특수성을 표로 확정한다.
2. 전체 7,342장에 EasyOCR을 적용할 수 있는 wide CSV 스키마를 만든다.
3. 가능하면 전체 OCR을 실행한다.
4. 전체 OCR 결과에서 날짜별/구간별 offset 규칙을 뽑는다.
5. real_bed_id 복구 manifest 초안을 만든다.

한 줄 결론:
Tesseract는 버리고 EasyOCR로 간다. 50장 결과는 다음 단계로 넘어갈 충분한 신호다. 다만 오늘의 핵심은 OCR 성능 향상이 아니라, 전체 7천 장에서 +64 offset, 0 offset, bed01 오염, 전환 날짜를 분리해내는 것이다.
